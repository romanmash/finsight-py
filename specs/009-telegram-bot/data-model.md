# Data Model: Telegram Bot (009)

## Overview

This model defines Telegram-facing interaction entities and state transitions required to authenticate users, process commands, throttle usage, format delivery, and send proactive notifications.

## Entities

### 1) TelegramIdentityBinding

- **Purpose**: Represents the linkage between Telegram sender identity and internal user account.
- **Core fields**:
  - `telegramHandle`
  - `userId`
  - `userActive`
  - `telegramChatId` (nullable until first successful interaction)
- **Validation rule**: Only active bindings are allowed to execute commands.

### 2) TelegramCommandEnvelope

- **Purpose**: Normalized incoming Telegram message used for routing.
- **Core fields**:
  - `messageId`
  - `fromHandle`
  - `chatId`
  - `rawText`
  - `parsedCommand` (nullable for free text)
  - `arguments[]`
- **Validation rule**: Commands requiring arguments must satisfy command-specific argument cardinality.

### 3) CommandExecutionOutcome

- **Purpose**: Deterministic result of command or free-text handling.
- **Core fields**:
  - `outcomeType` (`success|rejected|throttled|failed`)
  - `userFacingMessage`
  - `missionId` (optional when mission-producing)
  - `errorCode` (optional)
- **Validation rule**: Every handled input produces exactly one outcome envelope.

### 4) UserRateLimitWindow

- **Purpose**: Tracks per-user request counts in a configurable rolling or fixed window.
- **Core fields**:
  - `userId`
  - `windowKey`
  - `requestCount`
  - `windowExpiresAt`
- **Validation rule**: Requests over configured limit are denied until window reset.

### 5) FormattedDeliveryMessage

- **Purpose**: User-facing response after mission-label formatting and chunking.
- **Core fields**:
  - `label`
  - `fullText`
  - `chunks[]`
  - `chunkCount`
- **Validation rule**: Each chunk obeys Telegram size constraints while preserving order.

### 6) ProactivePushRequest

- **Purpose**: Notification dispatch request generated by internal workflows (alerts/briefs).
- **Core fields**:
  - `userId`
  - `message`
  - `sourceType` (`alert|daily_brief|system`)
  - `deliveryAttempted`
  - `deliveryResult` (`sent|skipped_no_chat|failed`)
- **Validation rule**: Missing chat destination results in `skipped_no_chat` with warning log.

## Relationships

- `TelegramIdentityBinding` authorizes `TelegramCommandEnvelope`.
- `TelegramCommandEnvelope` produces one `CommandExecutionOutcome`.
- `CommandExecutionOutcome` may reference mission entities from orchestration features.
- `UserRateLimitWindow` gates command processing before downstream execution.
- `FormattedDeliveryMessage` is generated from successful outcome payloads.
- `ProactivePushRequest` uses `TelegramIdentityBinding.telegramChatId` for delivery.

## State Transitions

### Command Handling Lifecycle

1. Receive message and parse envelope.
2. Resolve identity binding and active status.
3. Apply rate-limit gate.
4. Route command/free-text to corresponding backend action.
5. Format result and deliver response.
6. Persist chat destination if first successful contact.

### Proactive Push Lifecycle

1. Receive push request from internal event source.
2. Resolve user chat destination.
3. If missing destination -> skip + warn.
4. If present -> send message and record outcome.

### Throttle Lifecycle

1. Read/increment user window counter.
2. If within limit -> allow handling.
3. If exceeded -> return throttled outcome.
4. Reset eligibility after window expiry.
